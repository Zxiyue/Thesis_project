请在当前公开审计联邦学习项目基础上新增“多进程分布式真实通信实验版”。当前项目已有单进程 run.py，可完成 CNN 训练、Paillier 加密、Pedersen 承诺、KGC 掉线补偿、ModelSyncReq/ModelSyncResp、ModelSyncResp 的 CS 签名、真实 TCP 通信测量、Hardhat 链上 InitTx/FinalTx、runtime_cost.csv、communication_cost.csv、blockchain_cost.csv 等输出。现在不要破坏现有单进程 run.py，而是新增分布式模式。

一、总体目标

新增一个分布式实验版本，使 KGC、CS、Client 分别作为独立进程运行，并通过真实 HTTP/TCP 通信传递协议消息。

协议角色仍然只有：

KGC：初始化、密钥与掩码生成、上传验证、掉线补偿、模型一致性验证、FinalTx 生成与链上提交。

CS：联邦学习服务器，负责轮次推进、模型发布、收集上传、密文聚合、收集解密份额、恢复聚合明文、更新模型、广播模型。

Client：本地训练、模型同步、量化、加掩码、Paillier 加密、Pedersen 承诺、上传签名、生成解密份额。

BC：Hardhat/Solidity 智能合约公告板，记录 InitTx、FinalTx_r、FraudProofTx。

Verifier：最终读取链上声明和本地导出材料进行验证。

注意：不要把“控制器”设计成新的协议实体。控制器或启动脚本只能负责启动进程、向 CS 发送一次 StartExperiment 请求、收集日志、合并 CSV。控制器不能直接替客户端训练、不能替 CS 聚合、不能替 KGC 补偿或确认。

三、分布式角色设计

1. KGC 服务

新增 scripts/run_kgc_server.py。

启动示例：

python scripts/run_kgc_server.py --host 127.0.0.1 --port 9200 --config configs/mnist_iid_distributed.yaml

KGC 服务负责维护自己的状态，包括：

系统参数 sysPara、InitTx；
Paillier 密钥；
Pedersen 参数；
全局用户集合 U；
Uroot；
每轮 alpha；
每个客户端 mask 和 rho；
KGC 收到的 UpMsg 副本；
rootUp_r；
掉线集合 D_r；
Cdrop_r；
ComDrop_r；
上一轮 auditRoot；
当前 modelHash；
FinalTx_r。

KGC 服务至少提供接口：

POST /setup
POST /round_init
POST /upload_copy
POST /verify_uploads
POST /compensation
POST /final_confirm
GET /state
GET /health

接口职责：

/setup：生成系统参数、初始模型哈希、InitTx。可由 CS 在实验开始时调用，或由 start_experiment 向 KGC 调一次，但 start_experiment 不参与后续协议过程。

/round_init：由 CS 每轮调用，传入 r、modelHash_r、Uroot，KGC 生成 alpha_r、零和掩码 m_i,r、承诺随机数 rho_i,r，并向各客户端发送 MaskMsg，或把 MaskMsg 返回给 CS 转发。推荐 KGC 直接发送给 Client，这更符合方案。

/upload_copy：接收客户端发送给 KGC 的 UpMsg 副本。

/verify_uploads：验证上传签名和收据，构造有效上传集合 U_{r,1} 和 rootUp_r。

/compensation：根据 D_r=U_r\U_{r,1} 生成掉线补偿。若 D_r 为空，直接令 Cdrop_r 为 Paillier 同态单位元向量，ComDrop_r=1，不加密零向量。

/final_confirm：接收 CS 的 ModelMsg，验证模型哈希、模型差分和 ComAgg 一致性，生成 FinalTx_r，并调用智能合约 submitFinal 上链。如果 no-blockchain 模式开启，则只生成本地 FinalTx。

2. CS 服务

新增 scripts/run_cs_server.py。

启动示例：

python scripts/run_cs_server.py --host 127.0.0.1 --port 9300 --config configs/mnist_iid_distributed.yaml

CS 是轮次推进主体。CS 收到 StartExperiment 后，在 CS 进程内部执行训练轮次循环。不要让外部控制器逐步调度每个协议动作。

CS 服务保存状态：

当前全局模型 W_r；
modelHash_r；
各轮上传消息 UpMsg；
有效上传集合；
Craw_r、Cagg_r；
ComRaw_r、ComAgg_r；
CaggHash_r；
ShareMsg；
xAgg_r；
W_{r+1}；
modelHash_{r+1}；
sigAgg_r；
sigModel_r；
通信日志；
运行时间日志。

CS 服务至少提供接口：

POST /start_experiment
POST /init_model
POST /upload
POST /aggregate
POST /share
POST /model_sync
POST /publish_model
GET /state
GET /health

核心要求：

/start_experiment 是分布式实验的唯一轮次启动入口。收到后，CS 内部执行 for r in 1..R 的完整协议流程：

1. 请求 KGC /round_init；
2. 等待或确认客户端收到 MaskMsg；
3. 根据 dropout schedule 确定本轮在线客户端；
4. 通知在线客户端开始训练，或调用客户端 /train_upload；
5. 接收 Client->CS 的 UpMsg，同时客户端也会向 KGC 发送 UpMsg 副本；
6. 请求 KGC /verify_uploads 和 /compensation；
7. 根据 DropMsg 执行密文聚合和承诺聚合，得到 Cagg_r、ComAgg_r；
8. 对 AggMsg 签名并发送给在线客户端；
9. 接收客户端 ShareMsg；
10. 执行 Combine / 聚合密文恢复，得到 xAgg_r；
11. 检查 ComAgg_r = PedCom(xAgg_r;0)；
12. 更新模型得到 W_{r+1}；
13. 构造 ModelMsg，签名 sigModel_r，发送给 KGC /final_confirm；
14. KGC 通过后得到 FinalTx_r；
15. CS 只向本轮在线客户端发送 ModelBroadcast；
16. 掉线客户端不更新 local_model，下一轮上线时必须执行 ModelSyncReq/Resp。

/model_sync：处理客户端模型同步请求。若客户端 local_model_hash != modelHash_r，CS 返回当前模型 W_r、modelHash_r、modelBytesLength、modelBytesSha256，并加入 CS 签名 sigSyncCS。

签名 payload 为：

{
"r": r,
"client_id": client_id,
"modelHash": modelHash_r,
"modelBytesLength": len(model_bytes),
"modelBytesSha256": sha256(model_bytes).hexdigest()
}

sigSyncCS = Sig_CS(payload)

ModelSyncResp 至少包含：

{
"r": r,
"client_id": client_id,
"modelHash": modelHash_r,
"modelBytesLength": ...,
"modelBytesSha256": ...,
"modelBytes": ...,
"sigSyncCS": ...
}

3. Client 服务

新增 scripts/run_client_server.py。

启动示例：

python scripts/run_client_server.py --cid 1 --host 127.0.0.1 --port 9401 --config configs/mnist_iid_distributed.yaml

每个 Client 服务独立保存状态：

cid；
签名密钥；
本地数据分区；
local_model；
local_model_hash；
当前轮 mask；
当前轮 rho；
最新 alpha；
是否在线；
已发送 UpMsg；
解密份额状态。

接口至少包括：

POST /init_model
POST /receive_mask
POST /round_start
POST /train_upload
POST /receive_agg
POST /receive_model
POST /sync_model
GET /state
GET /health

关键要求：

/init_model：初始化客户端本地模型 W_0，并设置 local_model_hash=modelHash0。

/receive_mask：接收 KGC 的 MaskMsg，验证 KGC 签名和 alpha。

/round_start 或 /train_upload：客户端训练前必须检查：

local_model_hash == modelHash_r

如果不一致，客户端主动向 CS 发送 ModelSyncReq：

{
"r": r,
"client_id": cid,
"local_model_hash": local_model_hash,
"target_model_hash": modelHash_r,
"sigClient": optional
}

然后接收 CS 返回的 ModelSyncResp，验证：

1. modelBytesLength；
2. modelBytesSha256；
3. CS 签名 sigSyncCS；
4. H(W_r)==modelHash_r。

验证通过后更新 local_model/local_model_hash，再进行本地训练。

客户端训练必须使用自己的 local_model，不能使用 CS 或控制器传入的全局模型对象。

/train_upload：执行本地训练、模型差分、量化编码、加掩码、Paillier 加密、Pedersen 承诺、上传收据签名，生成 UpMsg。UpMsg 分别发送给 CS 和 KGC。

/receive_agg：接收 CS 的 AggMsg，验证 CS 签名，生成 ShareMsg 并发送给 CS。

/receive_model：接收轮末 ModelBroadcast，验证 sigModel 和 H(W_{r+1})==modelHash_{r+1}，更新 local_model/local_model_hash。只有本轮在线客户端会收到该消息。掉线客户端不能更新模型。

四、通信层要求

新增 fl_audit/distributed/http_client.py。

提供统一发送函数，例如：

send_json(sender, receiver, url, payload_type, payload, round_id, client_id=None)

要求记录真实通信：

round；
sender；
receiver；
link；
payload_type；
client_id；
bytes；
actual_seconds；
status；
ack_sha256；
receiver_host；
receiver_port；
send_start_time；
send_end_time。

如果使用 HTTP，actual_seconds 为请求开始到收到响应的时间。

服务端响应至少包含：

{
"ok": true,
"ack_sha256": sha256(received_payload_bytes).hexdigest(),
"server_receive_time": ...,
"server_response_time": ...
}

注意区分通信时间和计算时间。大消息接收接口建议先记录消息并 ACK，计算阶段单独通过 runtime_cost.csv 记录，避免把服务端计算时间混入通信时间。

如果为了简化，接口同步执行计算，也必须在日志中明确记录 process_seconds，不能把它误认为纯通信时间。

五、消息编码要求

新增 fl_audit/distributed/message_codec.py。

统一处理：

模型 state_dict 序列化；
大整数转字符串；
Paillier 密文向量序列化；
Pedersen 承诺序列化；
签名、收据、Merkle Root；
bytes/base64 编码。

要求所有网络通信的 payload size 基于真实序列化后的字节数计算。

ModelSyncResp、ModelBroadcast、ModelMsg 必须包含完整模型字节，不允许只用 modelHash 估算通信量。

UpMsg、AggMsg、DropMsg 必须包含真实密文向量或其真实序列化内容，不允许只用摘要代替通信量。

十、协议正确性要求

必须保证以下场景：

1. 第 3 轮 C2 掉线；
2. 第 3 轮结束时 C2 不收到 ModelBroadcast；
3. 第 4 轮 C2 重新上线；
4. C2 发现 local_model_hash != modelHash_4；
5. C2 发送 ModelSyncReq；
6. CS 返回带 sigSyncCS 的 ModelSyncResp；
7. C2 验证 sigSyncCS、modelBytesSha256、H(W_4)==modelHash_4；
8. C2 使用同步后的 local_model 训练；
9. communication_cost.csv 中出现：
   round=4, C2->CS, ModelSyncReq
   round=4, CS->C2, ModelSyncResp
10. run_trace.json 中记录：
    sync_performed=true
    sigSyncCS_verified=true
11. 最终 verify_result.csv 中 accepted=True。
